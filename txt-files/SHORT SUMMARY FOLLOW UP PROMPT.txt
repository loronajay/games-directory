The following architectural decisions are intentional and should not be redesigned unless explicitly requested.

Future solutions should respect these constraints.

Games Are Fully Isolated

Each game runs independently inside:

games/<slug>/

Games are TurboWarp exports and should not require manual editing.

All platform features are injected automatically through the patch system.

HTML Structure of Grid Cards Must Not Change

The grid page JavaScript depends on the exact structure:

<a class="game-card arcade-link" href="games/<slug>/index.html">

  <video muted loop preload="metadata">
    <source src="previews/<slug>.mp4" type="video/mp4">
  </video>

  <div class="game-title">Game Title</div>

  <div class="game-play-count">
    PLAYS: <span class="playCount">000000</span>
  </div>

</a>

Do not change these classes or structure without checking the existing JavaScript behavior.

Grid Cards Are Automatically Generated

Cards inside grid.html are generated by:

scripts/generate_grid.py

Between these markers:

<!-- AUTO-GENERATED-GAME-CARDS-START -->
<!-- AUTO-GENERATED-GAME-CARDS-END -->

Manual editing inside this section will be overwritten.

The Game Grid Must Ignore mini-arcade

The folder:

games/mini-arcade/

exists but must never appear in the grid.

This is intentional.

Preview Video Naming Convention

Preview videos use:

previews/<slug>.mp4

Example:

previews/apple-catcher.mp4
previews/x-pong.mp4

If a preview video is missing, the grid should still render the card with a black video area.

Metadata Overrides Are Optional

Each game can include:

games/<slug>/game.json

Example:

{
  "title": "Blade & Block",
  "order": 10,
  "card_classes": ["desktop-only"]
}

However:

• blank values should fall back to automatic defaults
• titles normally auto-generate from folder names

The system should not require manual metadata editing for normal games.

The Build Script Is the Source of Truth

The entire arcade is updated through:

scripts/build_arcade.py

This script performs:

ZIP import
metadata preservation/creation
HTML patching
grid regeneration
optional commit/push

Future solutions should integrate with the build pipeline, not bypass it.

Patch System Handles Platform Features

Game HTML should not be manually modified.

Instead, features are injected automatically by:

scripts/patch_all_games.py

This injects:

• mobile viewport tags
• jay-mobile.js controller
• GoatCounter analytics
• control overrides

The Goal Is a Scalable Arcade Frontend

The project is intentionally structured like an arcade operating system, not a typical website.

Design priorities:

• automated content ingestion
• automated UI updates
• minimal manual editing
• scalable game library
